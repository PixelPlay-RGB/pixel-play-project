# 아키텍처 및 폴더 구조 컨벤션 (SRP)

이 문서는 PixelPlay 프로젝트의 **단일 책임 원칙(Single Responsibility Principle)** 준수와 효율적인 코드 관리를 위한 폴더별 역할 및 가이드라인입니다.

---

## 1. 전역 구조 및 역할 (`src/`)

| 폴더/파일         | 역할                 | 가이드라인                                                                                   |
| ----------------- | -------------------- | -------------------------------------------------------------------------------------------- |
| **`actions/`**    | Server Actions       | 데이터 변경(Mutation), DB 접근, 보안이 필요한 로직 담당. 성공/실패 시 일관된 포맷 반환.      |
| **`app/`**        | Next.js App Router   | 라우팅 entry point. 페이지 레이아웃 정의 및 서버 사이드 데이터 페칭(Server Components) 수행. |
| **`components/`** | UI 컴포넌트          | 화면 렌더링 담당. 내부 로직은 최소화하고 필요한 경우 Hooks를 사용.                           |
| **`constants/`**  | 상수 관리            | API 엔드포인트, Query Key, 고정 메시지, 매직 넘버 등 정적 값 관리.                           |
| **`hooks/`**      | Custom Hooks         | 클라이언트 사이드 상태 로직, API 연동(TanStack Query), 반복되는 UI 로직 캡슐화.              |
| **`lib/`**        | 외부 라이브러리 설정 | Supabase, Zod 스키마, UI 유틸(cn) 등 외부 SDK 및 라이브러리 설정 관리.                       |
| **`mock/`**       | 목업 데이터          | 개발 및 테스트 단계에서 사용할 가상 데이터 및 관련 유틸리티.                                 |
| **`stores/`**     | 전역 상태 관리       | Zustand 스토어. 서비스 전역에서 공유되어야 하는 상태(인증 유저, 공통 UI 상태 등) 관리.       |
| **`types/`**      | 타입 정의            | 전역적으로 사용되는 TypeScript 인터페이스 및 타입 정의. (DB 자동 생성 타입 포함)             |
| **`utils/`**      | 도메인 유틸리티      | 특정 도메인(Auth, Chat 등)에 특화된 순수 함수 및 비즈니스 로직 유틸리티.                     |
| **`proxy.ts`**    | 라우트 가드/프록시   | 세션 검증 및 권한 기반 페이지 접근 제어 로직.                                                |

## 2. 상세 컨벤션 및 SRP 적용 지침

### [1] `src/app` (라우팅 및 데이터 공급)

- **Pages**: 비즈니스 로직을 직접 포함하지 않고, 서버에서 데이터를 가져와 하위 컴포넌트에 주입하는 역할에 집중합니다.
- **Layouts**: 공통 UI 구조를 정의하며, 상태를 직접 관리하기보다 `stores`나 `providers`를 통해 상태에 접근하도록 설계합니다.

### [2] `src/components` (UI 분리)

- **`ui/`**: shadcn 등 외부 라이브러리의 래퍼 컴포넌트. 프로젝트 전반의 공통 UI 규격 준수.
- **도메인 컴포넌트**: `auth/`, `chat/` 등 기능별로 구분하며, 복잡한 폼 로직이나 상태 전환은 `hooks/`로 위임합니다.

### [3] `src/lib` vs `src/utils`

- **`lib/`**: 외부 라이브러리를 프로젝트에 맞게 "설정"하는 곳입니다. (예: `supabase/client.ts`, `zod/auth.ts`)
- **`utils/`**: 프로젝트 내부의 비즈니스 로직을 보조하는 "순수 함수"들의 집합입니다. (예: `formatPhone`, `calculateRoomTime`)

### [4] `src/hooks` (관심사 분리)

- 컴포넌트에서 `useForm`, `useState`, `useEffect` 등을 사용하여 비즈니스 로직이 3줄 이상 길어질 경우 커스텀 훅으로 분리를 권장합니다.
- TanStack Query를 사용하는 로직은 반드시 `use-query-xxx.ts` 형태로 훅으로 관리합니다.

## 3. SRP 준수 여부 체크리스트

1. [ ] 컴포넌트 파일 내에 DB 접근 로직(supabase 호출 등)이 직접 포함되어 있는가? (→ `actions/`로 이동)
2. [ ] 컴포넌트 파일 내에 복잡한 정규식이나 데이터 포맷팅 로직이 있는가? (→ `utils/`로 이동)
3. [ ] 하나의 컴포넌트가 폼 관리, 데이터 페칭, UI 렌더링을 모두 수행하고 있는가? (→ `hooks/`로 로직 분리)
4. [ ] 전역적으로 반복 사용되는 상수값이 하드코딩 되어 있는가? (→ `constants/`로 이동)
