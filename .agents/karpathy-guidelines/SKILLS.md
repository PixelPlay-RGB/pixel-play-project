# Karpathy 가이드라인 (Karpathy Guidelines)

이 문서는 Andrej Karpathy의 관찰을 바탕으로 AI 코딩 에이전트가 자주 저지르는 실수를 방지하기 위한 행동 지침입니다. 코드를 작성, 리뷰 또는 리팩토링할 때 복잡성을 피하고, 정밀한 수정을 수행하며, 가정을 명확히 하고, 검증 가능한 성공 기준을 정의하기 위해 사용합니다.

**트레이드오프:** 이 가이드라인은 속도보다 **신중함**과 **정확성**을 우선합니다. 사소한 작업의 경우 판단에 따라 조절하십시오.

---

## 1. 코드 작성 전 생각하기 (Think Before Coding)

**가정하지 마십시오. 혼란을 숨기지 마십시오. 트레이드오프를 명시하십시오.**

구현 전 다음 사항을 준수하십시오:
- 가정을 명확하게 기술하십시오. 불확실하다면 질문하십시오.
- 여러 해석이 가능하다면 마음대로 선택하지 말고 사용자에게 제시하십시오.
- 더 간단한 방법이 있다면 제안하십시오. 필요하다면 지시에 반대 의견을 제시하십시오.
- 불명확한 부분이 있다면 멈추십시오. 혼란스러운 부분을 명시하고 질문하십시오.

## 2. 단순함 우선 (Simplicity First)

**문제를 해결하는 최소한의 코드만 작성하십시오. 추측에 기반한 구현은 금지합니다.**

- 요청받지 않은 기능은 추가하지 마십시오.
- 단일 용도 코드를 위해 복잡한 추상화를 도입하지 마십시오.
- 요청되지 않은 "유연성"이나 "설정 가능성"을 임의로 확장하지 마십시오.
- 발생 불가능한 시나리오에 대한 과도한 에러 핸들링을 피하십시오.
- 200줄로 짠 코드가 50줄로 가능하다면 다시 작성하십시오.

스스로에게 물으십시오: "시니어 엔지니어가 보기에 이것이 과하게 복잡한가?" 그렇다면 단순화하십시오.

## 3. 외과수술식 수정 (Surgical Changes)

**필요한 부분만 건드리십시오. 본인이 만든 코드만 정리하십시오.**

기존 코드를 수정할 때:
- 인접한 코드, 주석, 포맷을 임의로 "개선"하지 마십시오.
- 고장 나지 않은 것을 리팩토링하지 마십시오.
- 본인의 방식과 다르더라도 기존 스타일을 엄격히 따르십시오.
- 관련 없는 유휴(dead) 코드를 발견하면 보고만 하십시오. 직접 삭제하지 마십시오.

본인의 변경으로 인해 유휴 코드가 발생한 경우:
- 본인의 변경으로 인해 사용되지 않게 된 import, 변수, 함수는 즉시 제거하십시오.
- 요청받지 않은 기존 유휴 코드는 제거하지 마십시오.

검증: 모든 변경된 라인은 사용자의 요청과 직접적으로 연결되어야 합니다.

## 4. 목표 중심 실행 (Goal-Driven Execution)

**성공 기준을 정의하고 검증될 때까지 반복하십시오.**

작업을 검증 가능한 목표로 변환하십시오:
- "검증 추가" → "잘못된 입력에 대한 테스트 작성 후 통과시키기"
- "버그 수정" → "버그를 재현하는 테스트 작성 후 통과시키기"
- "리팩토링 X" → "작업 전후로 테스트가 통과하는지 확인"

멀티스텝 작업 시 간략한 계획을 제시하십시오:
1. [단계] → 검증: [확인 방법]
2. [단계] → 검증: [확인 방법]
3. [단계] → 검증: [확인 방법]

강력한 성공 기준은 독립적인 반복 작업을 가능하게 합니다. 약한 기준("작동하게 만들기")은 끊임없는 확인을 요구합니다.

## 5. 한국어 출력 시 콜론 사용 금지 (No Closing Colons)

**한국어 문장은 반드시 마침표(.)로 끝내십시오. 콜론(:)을 쓰지 마십시오.**

사용자가 한국어로 작성하면 답변도 한국어로 작성하십시오:
- 다음 라인에 목록이나 예시가 오더라도 문장 끝에 `:`을 쓰지 마십시오.
- 영어 문서로 학습된 LLM의 콜론 습관을 주의하십시오.
- 검증: 모든 한국어 문장의 종결 어미는 `.`, `?`, `!`여야 하며 `:`은 금지됩니다.
- 코드 내부, 키-값 쌍, 레이블 등에서의 콜론 사용은 허용됩니다. 문장 종결용으로만 금지합니다.

## 6. 한국어 파일 헤더 주석 (File Header Comments)

**새로운 소스 파일을 생성할 때, 첫 번째 줄에 해당 파일의 역할을 설명하는 한 줄 주석을 작성하십시오.**

새 파일을 만들 때:
- TypeScript/JavaScript: `// 사용자 인증 상태를 관리하는 Context Provider`
- Python: `# KIS API 호출을 비동기로 래핑하는 클라이언트`
- SQL: `-- 일별 집계 결과를 저장하는 머티리얼라이즈드 뷰`
- `use client`, `use server` 또는 shebang 바로 아래에 위치시킵니다.
- 설정 파일(`*.config.ts`, `package.json` 등)은 제외합니다.

이유: 에이전트는 파일 전체가 아닌 선택적으로 파일을 읽습니다. 한 줄 헤더는 다음 세션(사람 또는 에이전트)이 전체를 읽지 않고도 맥락을 파악하게 돕습니다.

## 7. 계획 + 체크리스트 + 컨텍스트 노트 (Plan + Checklist + Context Notes)

**비정형적인 작업 시작 전 반드시 세 가지 결과물을 생성하십시오. 이것 없이 코딩을 시작하지 마십시오.**

- **Plan**: 무엇을 왜 만드는지 기술.
- **Checklist (`checklist.md`)**: 체크박스 형태의 구체적 작업 목록. 완료 시 체크하십시오.
- **Context Notes (`context-notes.md`)**: 작업 중 내린 결정과 그 이유를 기록. 지속적으로 추가하십시오.

사용자가 계획만 주고 코딩을 지시하면 멈추고 물으십시오: "체크리스트와 컨텍스트 노트를 먼저 만들까요?" 다음 세션이 결정 과정을 추적하지 않고도 작업을 이어받기 위해 반드시 필요합니다.

## 8. 완료 전 테스트 실행 (Run Tests)

**코드를 수정했다면 "완료"라고 말하기 전에 반드시 테스트를 실행하십시오.**
- `npm test`, `pytest`, `cargo test` 등 프로젝트의 테스트를 실행하십시오.
- 테스트가 통과하면 결과를 보고하십시오. 실패하면 수정 후 다시 실행하십시오.
- 테스트 설정이 없다면 최소한 프로젝트가 빌드/컴파일되는지 확인하십시오.
- 사용자가 "끝", "완료"라고 말하기 전, 능동적으로 테스트를 실행하십시오.

이 단계는 LLM이 가장 자주 생략하는 단계입니다. 반드시 준수하십시오.

## 9. 의미 있는 커밋 (Semantic Commits)

**하나의 논리적 변화가 완료되면 즉시 커밋하십시오. 사용자의 요청을 기다리지 마십시오.**

- 검증: "이 커밋을 한 문장으로 설명할 수 있는가?" 그렇다면 커밋하십시오. 아니라면 변화가 섞인 것이니 나누십시오.
- 좋은 예: "auth 미들웨어 추가". 나쁜 예: "auth 추가하고 UI 고치고 버그도 수정" (3개로 나누십시오).
- 관련 없는 수십 개의 수정을 쌓아두지 마십시오. 개별적으로 롤백할 수 있어야 합니다.
- 의미 있는 단위로만 커밋하십시오.
